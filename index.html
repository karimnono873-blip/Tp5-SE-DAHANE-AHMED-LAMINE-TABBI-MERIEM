1. Simulations & Architecture Logicielle

Observez la différence entre un code bloquant (Séquentiel) et un OS Temps Réel (Concurrent) pour faire clignoter 3 LEDs à des vitesses différentes.

LED 1 (1000ms)
PA5

LED 2 (500ms)
PA6

LED 3 (400ms)
PA7

État du Système : Séquentiel (Bare Metal)

Durée de la boucle : 1000ms + 500ms + 400ms = 1900ms. Les LEDs sont bloquées les unes par les autres.

// SÉQUENTIEL (Mauvaise pratique pour la concurrence)
while(1) {
    HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);
    HAL_Delay(1000); // Bloque tout le processeur

    HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_6);
    HAL_Delay(500);

    HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_7);
    HAL_Delay(400);
}

Deux tâches (A et B) de même priorité tentent d'écrire sur l'écran I2C en même temps sans protection.

Race Condition (Collision de données)

// TÂCHE A
void vTaskA(void *pvParameters) {
    for(;;) {
        lcd_print("SYSTEM OK "); // Est interrompue au milieu !
        vTaskDelay(10);
    }
}

// TÂCHE B
void vTaskB(void *pvParameters) {
    for(;;) {
        lcd_print("ALERTE !!"); // S'incruste dans le bus I2C
        vTaskDelay(10);
    }
}

Utilisation d'un Mutex (Mutual Exclusion). Lorsqu'une tâche possède la "clé" (Mutex), les autres tâches sont mises en pause jusqu'à ce que le bus I2C soit libéré.

Accès Protégé (Synchronisation réussie)

SemaphoreHandle_t xMutexLCD;

void vTaskA(void *pvParameters) {
    for(;;) {
        // Demander la clé (Wait)
        if (xSemaphoreTake(xMutexLCD, portMAX_DELAY) == pdTRUE) {
            lcd_print("SYSTEM OK ");
            xSemaphoreGive(xMutexLCD); // Rendre la clé (Signal)
        }
        vTaskDelay(1000);
    }
}

2. Post-Lab : Ingénierie RTOS

1. Lancez la simulation et observez le comportement. Les LEDs respectent-elles maintenant leurs périodes respectives ?

Oui, avec FreeRTOS, les LEDs respectent rigoureusement leurs périodes de 1000 ms, 500 ms et 400 ms. Dans l'approche Bare Metal, l'utilisation de `HAL_Delay()` créait une exécution séquentielle où chaque délai bloquait l'exécution du reste du code[cite: 259, 264]. FreeRTOS gère les tâches de manière concurrente, permettant à chaque LED de clignoter indépendamment sans être retardée par les autres[cite: 270, 271, 304].

2. Que fait le processeur pendant qu'une tâche est en vTaskDelay ? Est-il bloqué ou peut-il exécuter une autre tâche ?

Pendant qu'une tâche est en `vTaskDelay()`, elle est placée dans l'état "Blocked" (bloquée)[cite: 305]. Cependant, le processeur **n'est pas bloqué**. L'ordonnanceur de FreeRTOS donne immédiatement le contrôle du CPU à une autre tâche prête à s'exécuter[cite: 305, 306]. Si aucune tâche n'est prête, le processeur exécute la tâche "Idle", optimisant ainsi l'utilisation des ressources.

3. Si vous modifiez la priorité de la Task 1 à 2 et laissez les autres à 1, quel est l'impact si les tâches demandent le processeur en même temps ?

FreeRTOS utilise un ordonnancement préemptif basé sur les priorités. Si la Task 1 a une priorité de 2 (plus élevée) et les autres une priorité de 1, la Task 1 **préemptera** toujours les autres tâches si elles demandent le processeur simultanément[cite: 307]. L'ordonnanceur donnera systématiquement le CPU à la Task 1 dès qu'elle sera prête, interrompant l'exécution des tâches de priorité inférieure.

4. Quel est, selon vous, le principal avantage de FreeRTOS concernant la maintenance du code si l'on veut ajouter une 4ème LED ?

Le principal avantage est la **modularité et l'indépendance** du code[cite: 308]. Pour ajouter une 4ème LED, il suffit de créer une nouvelle tâche (`vTaskLED4`) avec sa propre boucle et son `vTaskDelay()`, puis de l'ajouter à l'ordonnanceur avec `xTaskCreate()`. Contrairement à l'approche séquentielle où il faudrait recalculer tous les délais et réécrire la boucle principale complexe, l'ajout d'une nouvelle tâche n'impacte en rien le code ou le timing des tâches existantes.

5. Pourquoi voyez-vous des messages mélangés du type (Race Condition) sur l'écran LCD sans Mutex ?

On observe des messages mélangés car l'écran LCD est une **ressource partagée** accessible par les deux tâches (Tâche A et Tâche B) de même priorité[cite: 309, 311, 312, 313, 317]. Sans protection, si la Tâche A commence à envoyer "SYSTEM OK", l'ordonnanceur peut l'interrompre (préemption) au milieu de son écriture pour donner la main à la Tâche B, qui va alors écrire "ALERTE !!". Le contrôleur de l'écran reçoit un flux de données entrelacé, ce qui corrompt l'affichage.

6. Dans quel cas choisirait-on une Queue (File) plutôt qu'un Mutex pour gérer l'affichage LCD ?

On choisirait une Queue (File) si l'on souhaitait dédier une seule tâche **exclusivement** à l'affichage (une "Gatekeeper Task")[cite: 330]. Au lieu que chaque tâche utilise un Mutex pour accéder directement au LCD, les tâches enverraient leurs messages dans la Queue. La tâche d'affichage lirait ensuite ces messages séquentiellement dans la Queue et les afficherait proprement, évitant ainsi les blocages liés à l'attente d'un Mutex.

7. Pourquoi la tâche vTaskLog ne consomme-t-elle aucune ressource CPU tant qu'on n'appuie pas sur le bouton ?

La tâche `vTaskLog` ne consomme aucune ressource car elle utilise `xSemaphoreTake(..., portMAX_DELAY)`, ce qui la place dans l'état **"Blocked"** (bloquée) en attendant le sémaphore[cite: 334, 336]. Tant que le sémaphore n'est pas libéré (par l'appui sur le bouton via `xSemaphoreGive()`), l'ordonnanceur ignore complètement cette tâche lors de l'attribution du temps CPU[cite: 337, 338].

8. Pourquoi ne doit-on jamais oublier de faire un xSemaphoreGive() après un xSemaphoreTake() ?

Il est crucial de faire un `xSemaphoreGive()` car cela libère le sémaphore (ou le Mutex) pour qu'il puisse être utilisé par d'autres tâches[cite: 339]. Si on l'oublie, la ressource reste verrouillée. Toute autre tâche appelant `xSemaphoreTake()` sur cette même ressource restera bloquée indéfiniment. Cette situation de blocage permanent est appelée un **Deadlock (Interblocage)**, ce qui peut paralyser complètement le système.
